# Contributing to github-grep

Thanks for your interest in contributing! This guide covers everything you need to get started.

## Prerequisites

- [Node.js](https://nodejs.org/) >= 18
- [pnpm](https://pnpm.io/) >= 9
- A GitHub account (for testing with the API)

## Setup

```bash
git clone https://github.com/atinux/github-grep.git
cd github-grep
pnpm install
```

## Development Workflow

### Running in development

Use `pnpm dev` to run the CLI directly from TypeScript sources (via [tsx](https://github.com/privatenumber/tsx)) -- no build step needed:

```bash
pnpm dev -- unjs/citty
pnpm dev -- unjs/citty --state open
```

> Note the `--` separator between `pnpm dev` and the CLI arguments.

### Building

Compile TypeScript to JavaScript in `dist/`:

```bash
pnpm build
```

The compiled output is what gets published to npm. You can test it locally:

```bash
node bin/github-grep.mjs --help
```

### Running tests

```bash
# Run all tests once
pnpm test

# Watch mode (re-runs on file changes)
pnpm test:watch
```

Tests are written with [Vitest](https://vitest.dev/) and located in the `test/` directory.

## Project Structure

```
github-grep/
  bin/
    github-grep.mjs     # CLI entry point (plain JS with shebang, imports dist/)
  src/
    cli.ts               # CLI definition with citty, args parsing, orchestration
    fetcher.ts           # GitHub API logic: paginated issue + comment fetching
    writer.ts            # Markdown generation: YAML frontmatter, issue files, index
    types.ts             # TypeScript interfaces (Issue, IssueComment, CLIOptions, etc.)
  test/
    fixtures.ts          # Shared test data factories (makeIssue, makeComment, etc.)
    fetcher.test.ts      # Tests for API data transformation and helpers
    writer.test.ts       # Tests for Markdown generation and file output
  dist/                  # Compiled JS output (gitignored, generated by `pnpm build`)
```

### How it works

1. **`cli.ts`** parses arguments, resolves the GitHub token (flag > env var > `gh` CLI), validates inputs, and orchestrates the pipeline.
2. **`fetcher.ts`** uses [Octokit](https://github.com/octokit/octokit.js) to paginate through all issues for a repo, then fetches comments in parallel (bounded by `--concurrency`). Reactions come from the API response directly.
3. **`writer.ts`** converts each issue into a Markdown file with YAML frontmatter and writes it to disk. It also generates an `index.md` summary table.

## Writing Tests

- Use the factory helpers in `test/fixtures.ts` (`makeIssue`, `makeComment`, `makeReactions`) to create test data with sensible defaults.
- Pure functions (YAML generation, reactions formatting, etc.) are exported from `writer.ts` and `fetcher.ts` for direct unit testing.
- File I/O tests use `os.tmpdir()` and clean up after themselves.

Example:

```typescript
import { makeIssue, makeReactions } from "./fixtures.js"
import { buildIssueMarkdown } from "../src/writer.js"

it("includes comment reactions", () => {
  const issue = makeIssue({
    comments: [makeComment({ reactions: makeReactions({ "+1": 5 }) })],
  })
  const md = buildIssueMarkdown(issue)
  expect(md).toContain("**Reactions (5):** +1: 5")
})
```

## GitHub Token for Testing

The test suite does not call the GitHub API -- all API-dependent logic is tested via unit tests with mock data. You only need a token when running the CLI itself (`pnpm dev`).

Authentication is resolved automatically if you have the [GitHub CLI](https://cli.github.com/) installed and authenticated (`gh auth login`).

## Submitting Changes

1. Fork the repo and create a feature branch from `main`.
2. Make your changes.
3. Run `pnpm test` and make sure all tests pass.
4. Run `pnpm build` to verify the TypeScript compiles cleanly.
5. Open a pull request with a clear description of what you changed and why.
